🌐 Environment — self.env Deep Dive

Odoo ka har kaam self.env se shuru hota hai — samjho toh sab samajh aa jaayega

📖 What is the Environment?

Odoo mein self.env ek Environment object hai jo har model method mein available hota hai. Isme 4 cheezein hain — cursor (cr), user id (uid), context, aur superuser flag. Jab bhi tum koi model access karte ho, search karte ho, ya record banate ho — sab self.env ke through hota hai.

┌─────────────────────────────────────────────────────┐ │ self.env │ ├─────────────────────────────────────────────────────┤ │ .cr → Database cursor (PostgreSQL) │ │ .uid → Current user ID (integer) │ │ .user → Current user record (res.users) │ │ .context → Dict with lang, tz, active_id etc │ │ .company → Current company record │ │ .companies → All allowed companies │ │ .lang → Current language code ('en_US') │ │ .su → Boolean: is superuser? │ │ .registry → Model registry │ │ │ │ ['model'] → Access any model's recordset │ │ .ref() → Get record by XML ID │ └─────────────────────────────────────────────────────┘
💡 Key Insight: Environment is IMMUTABLE. Tum change nahi kar sakte — sirf with_context(), with_user(), sudo() se naya env ban sakta hai purane se.

Core Properties

Environment ke basic attributes

🗄️
self.env.cr
Database cursor — PostgreSQL se direct baat karta hai
CORE
▼
🗣️ Hinglish: cr = cursor = database connection. Isse tum raw SQL queries chala sakte ho. Lekin ORM methods prefer karo — raw SQL mein security, access rights, cache sab bypass ho jaata hai.
Python
12345678910111213141516171819
# ── Raw SQL query ──
self.env.cr.execute("""
    SELECT id, name, email
    FROM res_partner
    WHERE is_company = true
    LIMIT 5
""")
rows = self.env.cr.fetchall()
# → [(1,'ABC Corp','abc@x.com'), (2,'XYZ Ltd','xyz@y.com')]

# ── Parameterized query (SAFE from SQL injection) ──
self.env.cr.execute(
    "SELECT id FROM res_partner WHERE name = %s",
    ['Rahul']
)

# ── Dict result ──
self.env.cr.execute("SELECT id, name FROM res_partner LIMIT 3")
rows = self.env.cr.dictfetchall()
# → [{'id': 1, 'name': 'ABC Corp'}, ...]
⚠️ NEVER do this: cr.execute("SELECT * WHERE name = '%s'" % name) — SQL injection risk! Always use %s with params list.
💡 After raw SQL: Call self.env['model'].invalidate_model() to clear ORM cache, otherwise stale data dikhega.

CORE
▼

self.env.user
Current user ka full record (res.users)
CORE
▼
🗣️ Hinglish: uid sirf number hai, user pura record hai. Isse name, email, groups, company — sab access ho sakta hai.

self.env.context
Dictionary of contextual data — lang, tz, active_id, defaults
CORE
▼
🗣️ Hinglish: Context ek dict hai jo request ke saath travel karta hai. Isme language, timezone, active_id (konsa record open hai), aur tum custom keys bhi daal sakte ho. Bahut powerful hai — default values, behavior change, flags — sab context se hota hai.
Python
12345678910111213141516171819202122
# ── Read context ──
ctx = self.env.context
print(ctx.get('lang'))        # → 'en_US'
print(ctx.get('tz'))          # → 'Asia/Kolkata'
print(ctx.get('active_id'))   # → 42 (from form view)

# ── Default values via context ──
# In XML action:
#   context="{'default_partner_id': active_id}"
# In Python:
default_partner = ctx.get('default_partner_id')

# ── Check custom flag ──
if self.env.context.get('skip_validation'):
    return True

# ── Common context keys ──
# 'lang'           → Language code
# 'tz'             → Timezone
# 'active_id'      → Current record ID in view
# 'active_ids'     → Selected record IDs in list
# 'active_model'   → Current model name
# 'default_xxx'    → Default value for field xxx
⚠️ Context is FROZEN dict. self.env.context['key'] = val NAHI chalega. with_context() use karo naya context banane ke liye.

Model & Record Access

Models access karna aur XML ID se records lena

self.env['model.name']
Kisi bhi model ka empty recordset access karo
ENV
▼
🗣️ Hinglish: YEH SABSE IMPORTANT HAI. Kisi bhi model ko access karna ho — self.env['model.name'] likho aur empty recordset mil jaayega. Phir uspe search, create, browse — kuch bhi karo.
Python
1234567891011121314
# ── Access models ──
Partner = self.env['res.partner']
Product = self.env['product.product']
Order   = self.env['sale.order']

# ── Chain with ORM methods ──
partners = self.env['res.partner'].search([
    ('is_company', =, True)
])

# ── Even from inside another model ──
# Inside sale.order model:
products = self.env['product.product'].search([
    ('sale_ok', =, True)
])
🏢 Real-life: Sale order confirm karte waqt inventory check? self.env['stock.quant'].search([...]). Invoice create? self.env['account.move'].create({...}). Cross-model access hamesha self.env[] se hota hai.

self.env.ref(xml_id)
XML ID se seedha record fetch karo
ENV
▼
🗣️ Hinglish: Odoo mein har data record (views, groups, templates, demo data) ka ek XML ID hota hai jaise base.user_admin. ref() se seedha record le lo — search ki zarurat nahi.
Python
1234567891011121314
# ── Get specific records by XML ID ──
admin    = self.env.ref('base.user_admin')
usd      = self.env.ref('base.USD')
india    = self.env.ref('base.in')
template = self.env.ref('sale.email_template_edi_sale')

# ── Send email using template ──
template.send_mail(self.id)

# ── Safe: don't crash if not found ──
maybe = self.env.ref(
    'custom_module.some_record',
    raise_if_not_found=False
)  # → None if not found

Environment Switching

Context, user, company change — naya env banao

📋
.with_context(**kwargs)
Naye context ke saath naya environment
ENV
▼
🗣️ Hinglish: Existing context mein naye key-value add karne ho ya replace karna ho — with_context(). Original record nahi badalta, naya recordset milta hai naye env ke saath.
Python
123456789101112131415161718
# ── Add keys to existing context ──
new_rec = self.with_context(my_flag=True, lang='hi_IN')

# ── Default values for new records ──
self.env['sale.order'].with_context(
    default_partner_id=42,
    default_note='Auto-created',
).create({})

# ── Change behavior in your code ──
self.with_context(no_mail=True).action_confirm()

# Inside action_confirm:
def action_confirm(self):
    if not self.env.context.get('no_mail'):
        self._send_confirmation_email()

# ── Replace entire context ──
clean = self.with_context({})  # Empty context

🛡️
.sudo(flag=True)
Superuser mode ON/OFF
ENV
▼
🗣️ Hinglish: Access rights bypass karna hai — sudo(). Portal user order confirm nahi kar sakta normally, but backend code mein order.sudo().action_confirm() se ho jaayega. sudo(False) se wapas normal mode.
Python
12345678910111213
# ── Bypass access rights ──
order.sudo().write({'state': 'confirmed'})

# ── Create records that user can't normally ──
self.env['ir.logging'].sudo().create({
    'name': 'Custom Log',
    'type': 'server',
})

# ── Read data as sudo, then go back to normal ──
value = record.sudo().secret_field
# DON'T expose 'value' directly to user!
# Process it, then return safe data
⚠️ Security Rules:
1. sudo() sirf jahan zaroori ho
2. sudo() data directly user ko mat dikhao
3. sudo() ke baad wapas normal env mein kaam karo
4. User input pe sudo() KABHI mat lagao without validation

self.env.is_system()
Kya user Settings group mein hai?
UTILITY
▼
🗣️ Hinglish: is_admin() = superuser check. is_system() = "Settings" group (base.group_system) check. Admin hamesha system bhi hota hai, but system user admin nahi hota necessarily.
Python
1234
if self.env.is_system():
    # User has Settings/Technical access
    self._update_system_config()
# is_admin() ⊂ is_system() ⊂ normal user
